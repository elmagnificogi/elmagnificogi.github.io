# Agent

all in one，使用Agent辅助进行工作，如何协调和保障下面各个流程可以正常进行，并且大家同步

#### 背景
人员有：
- 产品经理
- 软件开发工程师
- 自动化测试工程师
- 总负责人



## 流程

#### 需求文档

wolai mcp获取产品需求文档内容（评审后的），然后给到代码内的agent进行读取和分析，给出工程设计文档，此时可以同步给出测试文档
工程文档和测试文档都同步回这个需求文档下



#### 技术文档

理论上技术文档出来以后，测试文档可以进行二次补全，把技术的一些边界或者隐含的一些问题进行测试
（如果是非技术或者是非熟悉项目，优先根据需求梳理技术文档，把需求可能会涉及的内容先总结出来）


#### 测试文档

测试文档经过上面两步，测试文档基本就全面了



#### 评审

需求、技术、测试文档都需要在此时评审结束，后续就是AI严格按照这个文档的内容进行开发，人工进行辅助

评审以后重新更新回上述文档中



#### 代码构建

代码构建，代码审核是人和Agent一起



#### 测试验证

增加自动化测试用例（代码构建同时开发），同时辅助人工测试进行流程验证

出现缺陷，根据情况流转回需求更新、工程设计、代码构建环节，重新执行各个步骤



#### 使用文档

更新用户的使用手册，功能文档



#### AI复核

开发完成以后最终AI复核测试、工程、需求文档是否一致，是否能达到互相印证的程度，是否可以直接交付给客户使用，是否可以合并回主分支，作为下一个迭代的基线


#### 多方交互

不同的软件，使用各自的Agent，Agent之间进行文档或者上下文的共享，从而获取到对方的设计逻辑，进而进行联调
- 默认以交互的协议作为共享边界



#### 缺少的环节

1.需求文档变更了，技术和测试需要通知变更，从而同步修改文档，这部分可以人工补齐流程或者用一个监控的Agent去做监控通知，然后重新评审（快速）
2.技术或者需求文档变更了，测试需要被通知变更，这部分可以人工补齐流程或者用一个监控的Agent去做监控通知，然后重新评审（快速）




#### 补充信息
1. 文档全部同步在wolai，暂时不与仓库产生任何关系，wolai有版本控制
2. 评审由总负责人最后拍板
3. 总体进度或者看板都由产品维护，需求不允许AI生产，允许AI审查需求
4. 代码构建就包含了CICD，给到测试验证就是一个完整产品了
5. 多需求并行时（如果不耦合、不冲突），代码需要从基线切不同分支后，同时开发，从而互不影响
